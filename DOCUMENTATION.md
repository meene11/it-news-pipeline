## 1. 프로젝트 설계 철학

### 1.1 왜 뉴스 데이터 파이프라인인가?

뉴스 데이터는 다음과 같은 특성 때문에 데이터 파이프라인 학습에 이상적입니다:

| 특성 | 이유 |
|------|------|
| **실시간 스트림** | 데이터가 지속적으로 생성되어 수집/처리/저장 파이프라인 전 과정을 경험 |
| **비정형 텍스트** | NLP, LLM 연동 경험을 얻을 수 있음 |
| **다중 소스** | 매체별 데이터 통합, 정규화 문제를 다룸 |
| **분석 가치** | 감성 분석, 경향성 탐색 등 비즈니스 인사이트 도출 가능 |
| **시각화 필요** | 분석 결과를 대시보드로 전달하는 End-to-End 경험 |

### 1.2 왜 단계적 버전 확장인가?

```
v1 (수집) → v2 (분석) → v3 (시각화/인사이트)
```

이 구조를 선택한 이유:

1. **관심사 분리(Separation of Concerns)**: 수집, 분석, 시각화를 독립된 모듈로 분리하여 각 계층의 책임을 명확히 함
2. **점진적 복잡도 증가**: 단순 HTTP 요청 → LLM API 연동 → 통계 집계 + 시각화 순으로 난이도를 높여가며 학습
3. **독립 배포 가능**: 각 버전이 이전 버전 위에 쌓이지만, 개별적으로도 동작할 수 있는 구조
4. **포트폴리오 스토리텔링**: 면접에서 "왜 이 구조로 설계했는가"라는 질문에 명확한 답변 가능

### 1.3 아키텍처 원칙

- **SDK보다 HTTP 직접 호출 우선**: Supabase Python SDK 대신 REST API를 직접 호출하여 HTTP 프로토콜 수준의 이해도를 높임
- **프레임워크 최소화**: React/Vue 대신 Vanilla JS로 웹 대시보드를 구현하여 DOM 조작과 비동기 통신의 기본기 확보
- **환경변수 기반 설정**: 하드코딩 방지, 12-Factor App 원칙 준수
- **방어적 프로그래밍**: 네트워크 실패, 파싱 오류 등을 try-catch로 처리하고 fallback 전략 적용

## 2. v1 상세 설계 및 구현

### 2.1 RSS 수집 전략

#### 왜 RSS인가?

| 방법 | 장점 | 단점 |
|------|------|------|
| **RSS** | 구조화된 XML, 법적 문제 없음, 무료 | 제공하지 않는 매체 존재 |
| 웹 스크래핑 | 모든 사이트 가능 | robots.txt 위반 가능, 구조 변경에 취약 |
| 유료 API (네이버 뉴스 API) | 공식 데이터, 안정적 | 일일 호출 제한, 비용 |

RSS를 선택한 이유:
- **법적 안전성**: RSS는 매체가 공개적으로 제공하는 피드이므로 크롤링 법적 이슈가 없음
- **구조화된 데이터**: XML 형식으로 파싱이 간단하고 안정적
- **무료**: API 키 없이 접근 가능
- **실제 매체 데이터**: 가상 데이터가 아닌 실제 IT 뉴스 사용

#### Fallback 전략

```python
rss_url = "https://feeds.feedburner.com/navernews/it"
fallback_urls = [
    "https://news.naver.com/main/rss/it.nhn",
    "https://rss.etnews.com/Section901.xml",
    "https://rss.zdnet.co.kr/zdnet/rss/news.xml",
    "https://feeds.feedburner.com/bloter",
]
```

하나의 RSS 소스에 의존하지 않고 **다중 소스를 순회**하며 수집을 시도합니다. 이유:

1. RSS 서버가 다운될 수 있음
2. 특정 매체가 RSS 제공을 중단할 수 있음
3. 네트워크 환경에 따라 특정 URL이 차단될 수 있음

이 패턴은 실무에서 외부 API를 호출할 때 흔히 사용하는 **Circuit Breaker** 패턴의 단순화된 버전입니다.

#### XML 파싱 -- 왜 ElementTree인가?

```python
import xml.etree.ElementTree as ET

root = ET.fromstring(res.content)
for item in root.iter("item"):
    title_el = item.find("title")
    link_el = item.find("link")
```

- `xml.etree.ElementTree`는 Python **표준 라이브러리**에 포함되어 있어 외부 패키지 설치가 불필요
- `feedparser` 같은 전문 RSS 파서도 있지만, XML 파싱의 기본 원리를 이해하기 위해 의도적으로 저수준 도구를 선택
- RSS 2.0 표준에서 `<item>` 태그 안의 `<title>`과 `<link>`만 추출하면 되므로 복잡한 파서가 불필요

#### 데이터 검증

```python
if title and link and len(title) > 5:
    news_items.append({
        "title": title.strip(),
        "url": link.strip()
    })
```

- `len(title) > 5`: 광고성 짧은 제목이나 RSS 파싱 오류로 인한 깨진 데이터 필터링
- `.strip()`: 양쪽 공백/개행 제거로 DB 데이터 품질 확보
- `news_items[:20]`: 한 번에 너무 많은 데이터를 저장하지 않도록 제한 (API 비용 제어, DB 부하 방지)

### 2.2 Supabase 연동

#### 왜 Supabase인가?

| 옵션 | 장점 | 단점 |
|------|------|------|
| **Supabase** | 무료 tier, PostgreSQL 기반, REST API 자동 생성, JS SDK 제공 | 무료 tier 용량 제한 |
| Firebase | Google 생태계, 실시간 DB | NoSQL이라 SQL 학습에 부적합 |
| MongoDB Atlas | 유연한 스키마 | 관계형 데이터 모델에 부적합 |
| SQLite | 로컬, 무설정 | 웹에서 직접 접근 불가 |
| AWS RDS | 프로덕션 급 | 비용, 설정 복잡 |

Supabase 선택 이유:
1. **PostgreSQL 기반**: 관계형 DB의 핵심(SQL, 스키마, 제약조건)을 학습할 수 있음
2. **REST API 자동 생성**: PostgREST가 내장되어 있어 별도 백엔드 서버 없이 DB에 직접 접근 가능
3. **JS SDK 제공**: 프론트엔드에서도 직접 DB를 조회할 수 있어 풀스택 경험 가능
4. **Row Level Security (RLS)**: 실무 수준의 보안 정책을 설정할 수 있음
5. **무료 tier 충분**: 500MB 스토리지, 월 2GB 전송량으로 학습 프로젝트에 충분

#### REST API 직접 호출 -- 왜 SDK를 쓰지 않았나?

```python
def supabase_request(method, table, data=None, params=None):
    url = f"{SUPABASE_URL}/rest/v1/{table}"
    headers = {
        "apikey": SUPABASE_KEY,
        "Authorization": f"Bearer {SUPABASE_KEY}",
        "Content-Type": "application/json",
        "Prefer": "return=representation"
    }
    return requests.request(method, url, headers=headers, json=data, params=params)
```

Python 코드에서 `supabase-py` SDK 대신 `requests`로 직접 호출한 이유:

1. **HTTP 프로토콜 학습**: 실제로 어떤 헤더가 필요하고, 어떤 HTTP 메서드를 사용하는지 이해
2. **PostgREST 이해**: Supabase의 REST API가 내부적으로 PostgREST를 사용한다는 것을 체감
3. **디버깅 용이**: SDK가 추상화한 부분에서 문제가 생겼을 때 HTTP 수준에서 원인 파악 가능
4. **의존성 최소화**: 외부 패키지 하나를 줄여 설치/배포가 간단해짐

반면 **웹 대시보드(index.html)에서는 Supabase JS SDK를 사용**한 이유:
- 브라우저 환경에서 CORS 처리, 실시간 구독 등이 SDK 없이는 복잡함
- Python과 JS 양쪽에서 다른 접근 방식을 사용해봄으로써 비교 학습

#### Supabase 인증 헤더 상세

```
apikey: SUPABASE_KEY         -- Supabase 프로젝트 식별
Authorization: Bearer KEY    -- PostgREST 인증 (anon key = 공개 읽기/쓰기)
Prefer: return=representation -- INSERT/UPDATE 후 결과를 응답 바디에 포함
```

- `apikey`와 `Authorization`에 동일한 키를 넣는 이유: Supabase가 Kong API Gateway를 거쳐 PostgREST로 전달하는 구조이기 때문
- `anon key`는 공개되어도 되는 키 (RLS 정책으로 접근 범위 제한)
- `service_role key`는 절대 클라이언트에 노출하면 안 됨 (RLS 우회)

### 2.3 터미널 대시보드 (rich)

#### 왜 rich인가?

- Python 터미널 UI의 사실상 표준 라이브러리
- 테이블, 패널, 컬러, 프로그레스 바 등을 간단한 API로 구현
- `print()` 대비 가독성이 월등히 높은 출력
- Will McGugan이 만든 오픈소스로, Textual의 기반 라이브러리

### 2.4 웹 대시보드 설계

#### CORS 프록시 -- 왜 필요한가?

```javascript
const CORS_PROXY = 'https://api.allorigins.win/raw?url=';
```

브라우저의 **Same-Origin Policy** 때문에 다른 도메인의 RSS 피드를 직접 fetch할 수 없습니다:

1. 브라우저 → RSS 서버: `Origin: http://localhost` 헤더 포함
2. RSS 서버: `Access-Control-Allow-Origin` 헤더 없음 → **CORS 차단**
3. 해결책: CORS 프록시 서버를 경유하여 서버 사이드에서 RSS를 가져옴

이것은 브라우저 보안 모델의 핵심 개념이며, 실무에서도 자주 만나는 문제입니다.

#### Vanilla JS 선택 이유

React, Vue 같은 프레임워크를 사용하지 않은 이유:
1. 이 프로젝트의 프론트엔드는 데이터 표시가 주 목적이므로 프레임워크가 과도한 엔지니어링
2. DOM API, fetch, async/await 등 **웹 기본기를 직접 사용**하여 기초 역량 증명
3. 번들러/빌드 도구 없이 HTML 파일 하나로 동작 → 배포 단순화
4. Supabase JS SDK의 `createClient`만으로 DB 연동 가능

## 3. v2 설계 -- AI 요약 + 감성 분석

### 3.1 왜 OpenAI gpt-4o-mini인가?

| 모델 | 가격 (1M tokens) | 한국어 성능 | JSON 모드 | 선택 이유 |
|------|-------------------|-------------|-----------|-----------|
| **gpt-4o-mini** | $0.15 input / $0.60 output | 우수 | 지원 | **비용 대비 성능 최적** |
| gpt-4o | $2.50 / $10.00 | 최상 | 지원 | 비용이 17배 높음, 이 용도에는 과분 |
| gpt-3.5-turbo | $0.50 / $1.50 | 양호 | 지원 | gpt-4o-mini보다 비싸고 성능 낮음 |
| Claude 3 Haiku | $0.25 / $1.25 | 우수 | 미지원(당시) | Structured Output 미지원 |
| Gemini Flash | 무료 tier | 양호 | 지원 | 한국어 뉴스 감성 분석 정확도 낮음 |

gpt-4o-mini 선택의 핵심 근거:
1. **비용 효율**: 뉴스 제목 요약은 짧은 입출력이므로 저렴한 모델로 충분
2. **Structured Output**: JSON 스키마를 강제할 수 있어 파싱 실패 없음
3. **한국어 성능**: 한국어 뉴스 제목의 뉘앙스를 잘 파악
4. **낮은 레이턴시**: 빠른 응답으로 배치 처리에 적합

### 3.2 Structured Output -- 왜 JSON 모드인가?

```python
# 예시: v2에서 사용할 프롬프트 구조
response = client.chat.completions.create(
    model="gpt-4o-mini",
    response_format={"type": "json_object"},
    messages=[
        {"role": "system", "content": """
        다음 뉴스 제목을 분석하여 JSON으로 응답하세요:
        {
            "summary": "한 줄 요약",
            "sentiment": "positive | negative | neutral",
            "confidence": 0.0~1.0
        }
        """},
        {"role": "user", "content": news_title}
    ]
)
```

일반 텍스트 응답 대신 JSON 모드를 사용하는 이유:

1. **파싱 안정성**: LLM이 자유 형식으로 응답하면 regex나 문자열 분리가 필요하고, 포맷이 변할 수 있음
2. **스키마 보장**: JSON 모드에서는 항상 유효한 JSON을 반환하므로 `json.loads()`가 실패하지 않음
3. **자동화 적합**: 프로그래밍적으로 값을 추출하여 DB에 저장하는 파이프라인에 필수
4. **일관성**: 매번 동일한 키 구조로 응답하여 후처리 로직 단순화

### 3.3 감성 분석 -- 왜 LLM 기반인가?

| 방법 | 장점 | 단점 |
|------|------|------|
| **LLM (gpt-4o-mini)** | 문맥 이해, 한국어 지원, 설정 간단 | API 비용, 레이턴시 |
| 사전 학습 모델 (KoBERT) | 무료, 빠름 | 파인튜닝 필요, 뉴스 도메인 특화 어려움 |
| 감성 사전 (KNU) | 무료, 빠름, 설명 가능 | 신조어/맥락 이해 불가 |
| 규칙 기반 | 투명, 빠름 | 유지보수 어려움, 커버리지 낮음 |

LLM 기반 감성 분석을 선택한 이유:
1. **문맥 이해**: "삼성 실적 하락에도 주가 상승" -- 사전 기반은 "하락"을 부정으로 판단하지만, LLM은 전체 맥락에서 긍정으로 판단
2. **한국어 뉴스 특화**: 뉴스 제목의 축약, 은유, 관용 표현을 이해
3. **confidence score**: 분석 결과에 대한 모델의 확신도를 함께 제공하여 신뢰도 평가 가능
4. **추가 학습 불필요**: 별도 학습 데이터 구축 없이 즉시 사용 가능

### 3.4 배치 처리 전략

```python
# 미분석 뉴스만 선별
unanalyzed = supabase_request("GET", "news_list", params={
    "select": "*",
    "summary": "is.null",
    "limit": "10"
})
```

- **WHERE summary IS NULL**: 이미 분석된 뉴스는 건너뜀 → API 비용 절약
- **LIMIT 10**: 한 번에 처리하는 양을 제한 → API Rate Limit 방지, 에러 시 손실 최소화
- **멱등성(Idempotency)**: 같은 스크립트를 여러 번 실행해도 중복 분석하지 않음

## 4. v3 설계 -- 트렌드 + 경향성 탐색

> **용어 참고**: 이 프로젝트에서는 "편향 분석"이 아닌 **"경향성 탐색"**이라는 표현을 사용합니다.
> "편향 판정"은 대규모 데이터, 복수 평가자, 통계적 유의성 검정이 필요하며,
> 이 프로젝트는 LLM 감성 분석 기반으로 매체 간 보도 톤 차이의 **경향성을 탐색하는 수준**입니다.

### 4.1 키워드 추출 -- 왜 LLM인가?

| 방법 | 장점 | 단점 |
|------|------|------|
| **LLM 엔티티 추출** | 문맥 이해, 의미 기반 추출 | API 비용 |
| konlpy (한국어 형태소 분석) | 무료, 빠름 | JVM 의존성, 고유명사 인식 한계 |
| TF-IDF | 통계 기반, 해석 가능 | 짧은 제목에서 정확도 낮음 |

LLM 선택 이유:
- 뉴스 제목은 평균 20~40자로 짧아서 통계 기반 방법의 정확도가 낮음
- "삼성전자", "갤럭시 S25", "AI 반도체" 같은 복합 키워드를 하나의 엔티티로 인식
- konlpy는 Java/JVM 설치가 필요하여 환경 설정이 복잡해짐

### 4.2 시각화 -- 왜 Chart.js인가?

| 라이브러리 | 크기 | 장점 | 단점 |
|------------|------|------|------|
| **Chart.js** | ~200KB | 간단한 API, 반응형, 풍부한 차트 타입 | 대규모 데이터에 느림 |
| D3.js | ~250KB | 최대 유연성 | 학습 곡선 높음, 저수준 API |
| ECharts | ~750KB | 풍부한 기능, 한국어 문서 | 번들 크기 큼 |
| Plotly | ~3MB | 인터랙티브 | 매우 큼 |

Chart.js 선택 이유:
1. **경량**: CDN으로 로드해도 부담 없는 크기
2. **간단한 API**: `new Chart(ctx, config)` 한 줄로 차트 생성
3. **프레임워크 무관**: Vanilla JS와 조합이 자연스러움
4. **반응형**: 별도 설정 없이 화면 크기에 맞춰 자동 리사이즈

## 5. 경향성 탐색 방법론 상세

### 5.1 미디어 편향(Media Bias)이란?

미디어 편향(Media Bias)은 뉴스 보도에서 특정 관점, 이념, 이해관계를 의식적 또는 무의식적으로 반영하는 것을 말합니다. 이 프로젝트에서는 이를 "판정"하는 것이 아니라, 매체 간 보도 톤 차이의 **경향성을 탐색**합니다.

#### 학술적 근거

이 프로젝트의 경향성 탐색은 다음 연구/프레임워크에 기반합니다:

| 근거 | 설명 | 적용 |
|------|------|------|
| **AllSides Media Bias** | 미국 미디어 편향 평가 사이트. 좌/중/우 5단계 분류 | 매체별 경향성 점수 산출의 참고 모델 |
| **MBFC (Media Bias/Fact Check)** | 미디어 편향과 사실 확인을 정량적으로 평가 | 경향성 점수 산출 방법론 참고 |
| **프레이밍 이론 (Framing Theory)** | Entman(1993) -- 동일 사건도 어떤 측면을 강조하느냐에 따라 수용자의 해석이 달라짐 | LLM을 통한 프레이밍 비교 분석의 이론적 근거 |
| **게이트키핑 이론** | 뉴스 편집자가 어떤 뉴스를 보도할지 선택하는 과정에서 경향성이 발생 | 매체별 키워드 빈도 차이 분석의 근거 |
| **감성 프레이밍** | 같은 사실이라도 긍정/부정 어조에 따라 인상이 달라짐 | 감성 분석 기반 경향성 탐색의 근거 |

#### 편향의 유형

| 편향 유형 | 설명 | 이 프로젝트에서의 측정 방법 |
|-----------|------|--------------------------|
| **보도량 편향 (Coverage Bias)** | 특정 이슈를 더 많이/적게 보도 | 키워드별 매체별 기사 수 비교 |
| **어조 편향 (Tone Bias)** | 같은 이슈를 긍정적/부정적으로 보도 | 매체별 감성 분포 비교 |
| **프레이밍 편향 (Framing Bias)** | 같은 이슈의 다른 측면을 강조 | LLM 프레이밍 분석 |
| **선택 편향 (Selection Bias)** | 어떤 뉴스를 보도할지 선택에서의 편향 | 매체별 키워드 분포 차이 |

### 5.2 경향성 점수(Bias Score) 산출 방법

#### 기본 개념

경향성 점수는 **특정 키워드/이슈에 대한 매체의 감성이 전체 평균에서 얼마나 벗어나는지**를 정량화한 값입니다.

#### 산출 공식

```
1단계: 매체별 감성 비율 계산
  매체 A의 키워드 K에 대한 감성 분포:
    P(positive) = 긍정 기사 수 / 전체 기사 수
    P(neutral)  = 중립 기사 수 / 전체 기사 수
    P(negative) = 부정 기사 수 / 전체 기사 수

2단계: 전체 평균 감성 비율 계산
  모든 매체의 키워드 K에 대한 평균:
    Avg(positive) = 전체 긍정 기사 / 전체 기사
    Avg(neutral)  = 전체 중립 기사 / 전체 기사
    Avg(negative) = 전체 부정 기사 / 전체 기사

3단계: 경향성 점수 산출
  Bias Score = (
    |P(positive) - Avg(positive)| +
    |P(neutral)  - Avg(neutral)|  +
    |P(negative) - Avg(negative)|
  ) / 2

  → 범위: 0.0 (편향 없음) ~ 1.0 (극단적 편향)
```

#### 예시

"AI 반도체" 키워드에 대해:

```
매체 A (5건): positive 4건(80%), neutral 1건(20%), negative 0건(0%)
매체 B (5건): positive 1건(20%), neutral 2건(40%), negative 2건(40%)

전체 평균 (10건): positive 50%, neutral 30%, negative 20%

매체 A Bias Score = (|80-50| + |20-30| + |0-20|) / 200 = 0.30
매체 B Bias Score = (|20-50| + |40-30| + |40-20|) / 200 = 0.30

→ 두 매체 모두 경향성 점수 0.30 (같은 이슈에 대해 반대 방향으로 편향)
```

#### /2로 나누는 이유

감성 비율의 합은 항상 1.0이므로, 절대 편차의 합의 이론적 최대값은 2.0입니다. 이를 2로 나누어 0~1 범위로 정규화합니다.

### 5.3 데이터 기반 -- 어떤 데이터를 사용하는가?

| 데이터 | 소스 | 용도 |
|--------|------|------|
| 뉴스 제목 + URL | RSS 피드 (v1 수집) | 분석 대상 |
| AI 감성 분석 결과 | OpenAI gpt-4o-mini (v2) | 편향 산출의 기반 데이터 |
| 매체 정보 (source) | RSS URL에서 도메인 추출 | 매체별 그룹핑 |
| 키워드 | LLM 엔티티 추출 (v3) | 이슈별 그룹핑 |

#### 경향성 탐색의 한계

1. **LLM 감성 분석 자체의 편향**: gpt-4o-mini도 학습 데이터의 경향성이 있을 수 있음
2. **제목만 분석**: 본문을 분석하지 않으므로 제목의 낚시성 여부를 반영하지 못함
3. **표본 크기**: 수집량이 적으면 통계적 유의성이 낮아짐
4. **시간 범위**: 특정 시점의 스냅샷이므로 장기적 편향 패턴을 놓칠 수 있음
5. **매체 수 제한**: 3개 매체만으로는 한국 IT 미디어 전체의 경향성을 대표하기 어려움

이러한 한계를 인지하고, 결과를 "절대적 편향 판정"이 아닌 "경향성 탐색"으로 해석해야 합니다.

### 5.4 프레이밍 분석 (LLM 기반)

동일 키워드에 대한 매체별 헤드라인을 LLM에 전달하여 프레이밍 차이를 분석합니다.

```python
# 예시 프롬프트
prompt = """
다음은 같은 이슈에 대한 서로 다른 매체의 뉴스 제목입니다.
각 매체가 어떤 관점/프레임을 강조하는지 분석하세요.

매체 A: "삼성전자 AI 반도체 투자 확대, 글로벌 1위 도약 기대"
매체 B: "삼성전자 무리한 AI 투자, 실적 악화 우려 커져"

분석:
- 매체 A: 성장/기회 프레임 (긍정적 전망 강조)
- 매체 B: 리스크/우려 프레임 (부정적 측면 강조)
- 차이점: 동일한 투자 결정을 기회 vs 리스크로 프레이밍
"""
```

## 6. 품질 전략 및 고도화 계획

### 6.1 현재 품질 수준 (v1)

| 항목 | 현재 상태 | 개선 필요 |
|------|-----------|-----------|
| 에러 처리 | try-catch로 기본 처리 | 재시도 로직, 로깅 시스템 필요 |
| 데이터 검증 | 제목 길이, 빈 값 체크 | URL 유효성, 중복 검사 강화 |
| 테스트 | 없음 | 단위 테스트, 통합 테스트 필요 |
| 보안 | .env로 키 관리 | anon key 클라이언트 노출 개선 |
| 모니터링 | 콘솔 로그만 | 구조화된 로깅, 알림 시스템 |

### 6.2 고도화 계획

#### 단기 (v2-v3 개발 중)

1. **데이터 품질 개선**
   - URL 중복 체크: Supabase `UNIQUE` 제약조건 또는 INSERT 전 SELECT
   - 제목 정규화: HTML 엔티티 디코딩, 특수문자 정리
   - 수집 시간 기록: RSS의 `pubDate` 파싱하여 원본 발행 시간 저장

2. **에러 처리 강화**
   - 지수 백오프(Exponential Backoff) 재시도 로직
   - API 실패 시 로컬 큐에 저장 후 나중에 재처리
   - 구조화된 로깅 (JSON 형식, 타임스탬프 포함)

3. **테스트 추가**
   - RSS 파싱 단위 테스트 (mock XML 데이터)
   - Supabase API 통합 테스트
   - AI 분석 결과 스키마 검증 테스트

#### 중기 (v3 이후)

4. **스케줄링 자동화**
   - cron job 또는 GitHub Actions로 정기 수집 자동화
   - 수집 → 분석 → 시각화 파이프라인 자동 실행
   - 실패 알림 (Slack, Discord webhook)

5. **성능 최적화**
   - RSS 수집 병렬화 (asyncio 또는 ThreadPoolExecutor)
   - OpenAI API 배치 호출 (여러 뉴스를 한 번에 분석)
   - Supabase 벌크 INSERT로 DB 호출 횟수 감소

6. **보안 강화**
   - RLS(Row Level Security) 정책 적용
   - API 호출 Rate Limiting
   - 입력값 서니타이징

#### 장기

7. **확장 가능한 아키텍처**
   - 수집기를 독립 마이크로서비스로 분리
   - 메시지 큐(Redis/RabbitMQ) 도입으로 수집-분석 비동기 처리
   - Docker 컨테이너화

### 6.3 품질 평가 방법

#### AI 분석 품질 평가

| 평가 방법 | 설명 | 적용 |
|-----------|------|------|
| **Human Evaluation** | 사람이 직접 AI 분석 결과를 검증 | 50건 샘플링하여 정확도 측정 |
| **Confidence Score 기반** | 낮은 신뢰도(< 0.6) 결과를 별도 검토 | 자동 플래깅 시스템 |
| **교차 검증** | 같은 뉴스를 다른 모델로 분석하여 결과 비교 | gpt-4o-mini vs gpt-4o 비교 |
| **시간에 따른 일관성** | 비슷한 뉴스가 일관된 감성 결과를 내는지 확인 | 주기적 모니터링 |

#### 파이프라인 품질 지표 (KPI)

| 지표 | 측정 방법 | 목표 |
|------|-----------|------|
| **수집 성공률** | 성공 수집 수 / 시도 수 | > 95% |
| **데이터 완전성** | NULL 필드가 없는 레코드 비율 | > 99% |
| **AI 분석 정확도** | Human evaluation 일치율 | > 80% |
| **분석 커버리지** | 분석 완료 뉴스 / 전체 뉴스 | > 95% |
| **레이턴시** | 수집 → 분석 완료까지 소요 시간 | < 5분 |
| **가용성** | 파이프라인 정상 가동 시간 | > 99% |

#### 경향성 탐색 품질 평가

1. **통계적 유의성 검정**: 매체 간 감성 차이가 우연이 아닌지 카이제곱 검정으로 확인
2. **표본 크기 확인**: 각 매체-키워드 조합의 최소 기사 수 기준 설정 (예: 5건 이상)
3. **시간 윈도우**: 경향성 점수를 일별/주별로 산출하여 추세 변화 모니터링
4. **교차 검증**: LLM 감성 분석 결과와 감성 사전 기반 분석 결과를 비교
